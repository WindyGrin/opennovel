# 辛辣审查

以刻薄编辑+暴躁读者+同行作者三重人格，对章节执行不留情面的文学水准批判。

## 快速参考

| 参数 | 说明 |
|------|------|
| `--range N-M` | 多章审查 |
| `--full` | 全维度轰炸（默认只打2-3个要害） |
| `--tag-only` | 只更新标签库 |

**产出**: `审查报告/第X章毒舌审查.md` + 更新 `设定集/毒点标签库.md`

## 核心约束

- 必须用"你"直接对作者说话
- 每个崩点引用≥1句原文作证据
- 必须有根因分析，不许泛泛而骂
- 涉及设定须标注 `📖 设定参考:` + 文件位置
- **标签库是起点，不是牢笼**：毒点标签库提供参考词汇，但审查的核心判断力来自阅读体验。如果一个毒点标签库里没有现成标签——造一个。标签库的意义是每次审查后变丰富，不是每次审查前划定批评范围。教条式对照标签库找崩点 = 用体检报告写诊断书，本末倒置
- **大纲是草稿，章节是成品**：如果章节内容精彩有新意，但偏离了大纲——批评方向是"大纲过时了，该更新"，不是"章节写错了，该砍"。好章节让大纲臣服

## 执行清单

### Step 0: 预检

检查章节文件存在：`python .opencode/scripts/webnovel.py chapter-path --chapter {N}`

### Step 1: 加载上下文

```
- 章节正文
- 大纲/卷纲/第1卷/详细大纲.md （查找本章条目）
- 设定集/毒点标签库.md
- 前两章结尾（各取最后50行）
```

### Step 2: 快速扫读（读者视角）

一口气读完，记录：哪里走神？哪里想跳过？哪里觉得尬？读完最大感受？

### Step 3: 要害定位

凭阅读直觉识别最崩的点，不是凭标签库。

先问自己的阅读感受：
- 哪里读着读着走神了？
- 哪段让你下意识想划走？
- 哪里读完觉得尬/假/空？

然后再归类（文笔/情节/角色/叙事/结构/设定）。标签库可以提供术语，但不能替你判断——如果你发现一个问题标签库里没有，那就是新标签的种子。

### Step 4: 深度解剖

> 📖 加载 `references/critique-framework.md` 获取完整解剖框架

对每个崩点：定位原文 → 判定根因（写法/设计/认知） → 搜索设定集验证

### Step 5: 生成三段式报告

> 📖 加载 `references/report-template.md` 获取报告骨架+毒度标尺

### Step 6: 更新标签库

> 📖 加载 `references/tag-schema.md` 获取标签格式

审查结束后的自然动作——本次发现的崩点中，哪些值得成为标签留给下次参考？

- **发现的模式标签库没有** → 创建新标签，标注首现章
- **标签库已有的被触发了** → 更新"最近出现"，补充新案例（如果新案例比旧的更有代表性则替换）
- **慎用标签匹配**：不要为了凑标签而扭曲批评角度。同样的原文现象在不同章里崩的原因不同——标签是近似归类，不是精确诊断

### Step 7: 落库

写 `审查报告/第{CHAPTER}章毒舌审查.md`，更新 `设定集/毒点标签库.md`
