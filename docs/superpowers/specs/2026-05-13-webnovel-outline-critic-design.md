# webnovel-outline-critic 设计规格

> 创建日期: 2026-05-13
> 状态: 待实现
> 依赖: webnovel-critic (参考模式)

## 1. 问题陈述

现有 `webnovel-critic` 和 `webnovel-review` 仅审查章节正文。总纲、卷纲的结构性质量缺乏专用审查机制。大纲层面的缺陷（伏笔孤儿、卷间断层、节拍密度塌方）只能在章节写完之后通过回溯发现——此时修复成本已高。

## 2. 目标

创建一个毒舌三段式大纲审查skill，镜像 `webnovel-critic` 的审查管线，聚焦总纲和卷纲的结构性质量分析。支持快速单文档审查和深度跨文档验证两种模式。

## 3. 架构

```
.opencode/skills/webnovel-outline-critic/
├── SKILL.md                       # 主工作流，含mode分支gate
├── references/
│   ├── critique-framework.md      # 大纲专属解剖框架
│   ├── report-template.md         # 三段式报告模板
│   ├── quick-checklist.md         # quick模式6维速查清单
│   ├── deep-checklist.md          # deep模式跨文档对照清单
│   └── tag-schema.md              # 大纲缺陷标签格式
└── (无scripts/)                    # 纯LLM逻辑
```

文件规模上限：SKILL.md ~150行，每个reference ~60行。

## 4. 模式Gate（Step 0-1）

### 4.1 参数解析

| 参数 | 说明 | 默认 |
|------|------|------|
| `--mode quick|deep` | 审查深度 | quick |
| `--target <总纲\|卷N\|全卷>` | 审查目标 | 总纲 |

### 4.2 加载Gate

**关键约束**：quick模式绝不加载设定集文件。Gate写死在SKILL.md中，不允许"顺便看看"。

```
Step 1: 模式Gate
  if mode == "quick":
    加载列表:
      - 总纲.md (必加载，~90行)
      - 设定集/大纲缺陷标签库.md (索引表，~25行)
      - if target是卷N: 该卷5个卷纲文件
      - if target是卷N: 上一卷的卷骨架.md + 时间线.md (衔接锚点)
      - if target是卷1: 跳过上一卷(无)
    上限: ~8文件, ~25K token

  if mode == "deep":
    执行quick全部加载 +
      - 所有已规划卷的卷骨架 + 时间线 + 节拍表
      - 设定集/核心设定/世界观.md
      - 设定集/核心设定/力量体系.md
      - 总纲的伏笔表(18条)
      - 目标卷首/卷中/卷末各2章正文结尾抽样(各70行)
    上限: ~18文件, ~75K token
```

deep的额外分析在quick分析完成后作为独立section追加。两者隔离——quick阶段不碰deep文件。

## 5. Quick模式：6维速查清单

不加载设定集的前提下，对大纲文件做6个维度的检查：

| # | 维度 | 检查方法 | 依赖文件 |
|---|------|---------|---------|
| 1 | 结构完整度 | 卷纲5文件是否齐全；每章条目14字段(目标/阻力/代价/时间锚点/爽点/Strand/反派层级/视角/关键实体/本章变化/章末未闭合问题/钩子/章内时间跨度/与上章时间差)是否缺漏 | 目标卷纲全部 |
| 2 | 因果链 | 章间"目标→阻力→代价→钩子"是否形成闭环；钩子是否被下章承接 | 详细大纲 |
| 3 | 节奏密度 | 爽点类型是否机械重复；Strand分配(Quest/Fire/Constellation)是否失衡 | 详细大纲 + 节拍表 |
| 4 | 时间锚点 | 章间时间差是否合理；倒计时是否一致；无跨章天数算术错误 | 时间线 |
| 5 | 衔接质量 | 卷首是否承接上卷末；卷末钩子在下一卷是否兑现 | 卷骨架 + 上一卷卷骨架 |
| 6 | 红牌规则 | 是否违反总纲创作基调约束(如"禁止倒吸凉气/跪拜") | 总纲创意约束节 |

每个低分维度展开三段式解剖。目标产出3-5个崩点。

**目标适配**：
- `target=总纲`：只检查维度5(衔接质量，检查卷划分逻辑)和维度6(红牌规则)，外加总纲专属检查(创意约束完整性、主角成长线是否有节点、反派分层是否每层有对应卷)。维度1-4不适用。
- `target=卷N`：全部6个维度适用。
- `target=全卷`：对每卷独立运行6维度，再汇总。

## 6. Deep模式：4项跨文档对照检查

| # | 检查项 | 对照源 | 验证逻辑 |
|---|--------|--------|---------|
| D1 | 总纲↔卷纲一致性 | 总纲(反派分层/核心冲突/成长线) ↔ 全卷纲 | 总纲承诺的反派是否在对应卷纲中出场；主角关键跃迁节点是否在卷纲中有对应章 |
| D2 | 伏笔表闭环 | 总纲(伏笔表18条) ↔ 全卷详细大纲 | 每条伏笔的埋设章/回收章在大纲中是否真实存在条目；孤儿伏笔(只埋不收)/幽灵伏笔(只收不埋) |
| D3 | 卷间节拍密度 | 全卷节拍表 + Strand分布 | 卷间Strand比例(Q/F/C)是否合理；爽点规划的卷末大爽点是否有对应高潮章 |
| D4 | 大纲↔正文兑现率 | 详细大纲 ↔ 正文抽样 | 大纲规划的核心爽点在正文中是否兑现 |

deep报告在quick的3-5崩点之上追加4-10个跨文档崩点，外加一张跨文档对照表。

## 7. 报告格式

继承 `webnovel-critic` 的三段式报告模板，追加deep模式专属section：

```
═══════════════════════════════════════
    大纲毒舌审查 · 总纲/第N卷「名称」
═══════════════════════════════════════
📊 综合毒度: X.X/10 ｜ 崩点: N个 ｜ 模式: quick/deep
   ▶ 公式: (Σ调整后崩点分²)/(Σ调整后崩点分) × 密度系数

⚠️ 惯性警告: [标签名]已连续第N次触发 (if applicable)

── Quick审查 ──
🔪 崩点一：[标签名]  毒度 X/10 [档位]
① 开骂 ② 解剖 ③ 改法
... (3-5个崩点)

── Deep审查 (仅--mode deep) ──
🔪 崩点D1：[标签名]  毒度 X/10 [档位]
... (4-10个跨文档崩点)

📊 跨文档对照表 (仅--mode deep)
| 总纲条目 | 对应卷纲 | 状态 | 偏差 |
|---------|---------|------|------|
| 沈崇武Ch31发难 | 第1卷详细大纲 | ❌ | 实际Ch31为铁骊决定 |

📋 次生问题速览
═══════════════════════════════════════
```

## 8. 标签库

### 8.1 位置

```
设定集/大纲缺陷标签库.md              ← 速查索引(轻量，~25行)
设定集/大纲缺陷标签库/<标签名>.md      ← 完整条目(按需加载)
```

### 8.2 索引表

| 类别 | 标签 | 毒度基准 | 触发条件（摘要） | 惯性 |
|------|------|:------:|------|:----:|
| 结构 | 卷纲文件缺失 | 8 | 目标卷5文件不全 | 0 |
| 结构 | 条目字段漏填 | 6 | 章条目14字段缺≥3项 | 0 |
| 逻辑 | 因果链断裂 | 8 | 章间目标→钩子不闭环 | 0 |
| 逻辑 | 伏笔孤儿 | 9 | 伏笔埋设/回收章在大纲中不存在 | 0 |
| 逻辑 | 时间线矛盾 | 7 | 倒计时/章间天数算术矛盾 | 0 |
| 节奏 | Strand失衡 | 7 | 近10章某Strand占比超上限2倍 | 0 |
| 节奏 | 爽点密度塌方 | 8 | 连续5章无任何爽点标记 | 0 |
| 节奏 | 节拍机械重复 | 6 | 同类型爽点连续≥4次 | 0 |
| 一致性 | 大纲设定矛盾 | 9 | 总纲承诺与卷纲执行不匹配 | 0 |
| 一致性 | 大纲正文偏离 | ※ | 同大纲脱节规则 | 0 |
| 一致性 | 卷间角色断连 | 7 | 核心角色跨卷消失≥1卷 | 0 |
| 规划 | 大纲架空 | 8 | 大纲爽点/冲突在正文抽样中0兑现 | 0 |
| 规划 | 钩子断裂_大纲级 | 7 | 卷末钩子在下一卷卷首无承接条目 | 0 |
| 规划 | 倒计时崩坏 | 6 | 倒计时天数跨卷不一致 | 0 |

### 8.3 子文件格式

与 `毒点标签库` 子文件格式完全一致：类别/触发条件/毒度基准/档位锚点(轻/中/重)/首次记录/最近出现/惯性计数/典型案例。

### 8.4 加载策略

审查Step 3定位崩点时：先扫索引表识别模式 → 按需加载命中的子文件确认档位锚点匹配。与webnovel-critic完全一致的流程。

## 9. 审查管线

Step 0: 预检 → 解析参数，检查目标文件存在性
Step 1: 模式Gate → 按quick/deep加载文件（硬分支，不可膨胀）
Step 2: 快速扫读 → 一口气读完所有已加载文件，记录结构直觉
Step 3: 要害定位 → 凭阅读直觉识别最崩的点，扫标签库索引
Step 4: 深度解剖 → 定位原文→档位判定→根因分类→设定引用
Step 5: 生成报告 → 三段式+deep对照表(如deep模式)
Step 6: 更新标签库 → 新标签创建/已有标签惯性+1/更好案例替换
Step 7: 落库 → `审查报告/大纲/总纲毒舌审查.md` 或 `审查报告/大纲/第N卷毒舌审查.md`

## 10. 触发条件

| 用户表述 | 解析 |
|---------|------|
| "审查大纲"/"审总纲" | `--mode quick --target 总纲` |
| "审查第N卷大纲" | `--mode quick --target 卷N` |
| "深审大纲"/"全面审查大纲" | `--mode deep --target 全卷` |
| "/webnovel-outline-critic --deep" | 同上 |

## 11. 产出

| 模式 | 产出 |
|------|------|
| quick | `审查报告/大纲/<目标>毒舌审查.md` + 标签库更新 |
| deep | 同上 + 跨文档对照表 |

## 12. 非目标

- 不审查章节正文（由webnovel-critic负责）
- 不审查设定集（后续独立skill）
- 不修改大纲文件（只出报告，不出修改）
- 不依赖外部Python脚本
